当我设计一个智能体系统时，通常不会先从 prompt 开始，而是先画任务结构。

<!--more-->

如果任务天然就是多步骤的，那么最先要明确的是：哪些步骤应该交给模型，哪些步骤应该交给确定性逻辑。

我会先拆成下面四层：

## 1. Intent Layer

这一层负责识别用户到底想完成什么。不是每个输入都足够明确，所以这里要容忍模糊，并且能补足上下文。

## 2. Planning Layer

这里负责把目标拆成步骤。一个好的计划层，不只是列出步骤，还会明确每一步依赖什么信息、调用什么工具、失败后怎么办。

## 3. Execution Layer

执行层尽量少依赖“猜”。能用规则做的事情，不交给模型。能用 API 确认的结果，不靠语言生成来假设。

## 4. Review Layer

这是很多 demo 最容易缺失的一层。系统要知道输出有没有达标，格式对不对，信息全不全，是否需要再次尝试。

我尤其重视下面这条原则：

> 不要把“可观察性”留到最后再补。

Agent 一旦进入真实工作流，没有日志、状态和反馈，你很快就不知道它到底是“没做”、"做错了"，还是“做了一半卡住了”。

一个最小但实用的 Agent 工作流，至少应该能回答这些问题：

- 当前执行到了哪一步
- 上一步用了什么输入
- 工具是否返回异常
- 结果为什么被判定为成功或失败

只有当这些基础设施存在时，Agent 才会从实验性功能变成可以迭代的系统。
