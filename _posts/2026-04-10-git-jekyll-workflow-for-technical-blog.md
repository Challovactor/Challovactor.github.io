很多人会把博客站点搭起来，但坚持更新很难。问题往往不在页面，而在写作工作流不顺。

<!--more-->

我更推荐把博客系统做成一个尽量简单的工程系统：仓库清晰、写作路径短、发布零额外负担。

## 1. 仓库怎么建

如果你是个人主页，最简单的仓库名是：

```text
yourname.github.io
```

这样 GitHub Pages 会直接把仓库当作站点根目录。

优点是：

- 域名规则清晰
- 发布链路最短
- 不需要额外配置子路径

## 2. 写作目录怎么组织

我推荐保留下面这些目录：

```text
_posts/
_layouts/
blog/
tags/
categories/
search/
```

其中最重要的是 `_posts`。

只要文件命名正确，Jekyll 就会自动生成文章页。

## 3. 一篇文章至少要写哪些 front matter

我建议每篇文章最少包含这些字段：

```yaml
---
title: 文章标题
date: 2026-04-10 10:00:00 +0800
post_type: Tutorial
categories:
  - 工具使用
tags:
  - GitHub Pages
  - Jekyll
reading_time: 6 min read
summary: 一句话摘要
---
```

这样做的好处是：

- 列表页能显示摘要
- 标签页和分类页更清晰
- 搜索页能更准确命中

## 4. 我建议的写作流程

我的流程很简单：

1. 在 `_posts` 新建 Markdown 文件。
2. 先写 front matter。
3. 第一段写问题背景。
4. 在 `<!--more-->` 之后展开正文。
5. 提交到 Git 仓库并推送。

这样你不会在“先写完再整理格式”和“先做页面再写内容”之间来回切换。

## 5. 技术博客最容易出的问题

我见过最常见的几个问题是：

- 页面搭得很漂亮，但文章太少
- 分类很多，但没有真正区分内容结构
- 发布步骤太重，导致更新频率很低
- 文档链接和图片路径经常失效

所以我更推荐一种保守策略：

- 模板简单
- 分类少而清晰
- 每篇文章能独立成立
- 发布流程尽可能自动

## 6. 一篇教程型文章怎么写更有效

教程不要一上来讲原理，而应该先回答：

1. 这篇文章解决什么问题
2. 读完后你能得到什么
3. 环境和前置条件是什么
4. 最短成功路径是什么

如果这四点没有写清楚，读者通常很难坚持读下去。

## 7. 为什么我喜欢这种博客方案

GitHub Pages + Jekyll 不一定是最现代的方案，但它足够稳定、轻量、容易长期维护。

对技术博客来说，这比“功能很多”更重要。

真正决定你博客质量的，通常不是站点框架，而是：

- 你是否持续更新
- 文章是否结构清晰
- 内容是否来自真实实践

如果一个博客能让你每周稳定写一篇，它就已经是一个好系统了。
